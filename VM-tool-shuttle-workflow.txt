
VM Tool Shuttle Workflow
========================

Purpose
-------
Design and maintain a portable AMD64/x86_64 virtual machine workflow that:

- Runs natively on a Debian Linux virtualization host
- Can be exported/imported into VMware ESXi
- Uses portable virtual hardware and disk formats
- Supports future backup and archival workflows
- Minimizes hypervisor lock-in

Recommended Virtualization Stack
--------------------------------

Primary Hypervisor Platform on Debian:
- QEMU
- KVM
- libvirt
- virt-manager

Reasoning:
- Linux-native
- Enterprise-proven
- Excellent ESXi interoperability
- Strong tooling ecosystem
- Easy disk conversion workflows

Avoid:
- Building the primary VM in VirtualBox
- Hypervisor-specific guest additions
- Experimental virtual hardware

Target Architecture
-------------------

Debian Host
    ↓
KVM/QEMU Virtual Machine
    ↓
Portable Disk Format (qcow2)
    ↓
Convert to VMware VMDK
    ↓
Import into VMware ESXi

Recommended VM Design Standards
--------------------------------

Firmware:
- Use UEFI only
- OVMF firmware under KVM
- Avoid legacy BIOS

Machine Type:
- q35

CPU:
- host-model or conservative x86_64 model
- Avoid exotic CPU flags

Disk Bus:
- VirtIO

Network Adapter:
- VirtIO

Disk Partitioning:
- GPT partition table

Guest Filesystem:
- ext4 or XFS

Recommended Guest Operating Systems
-----------------------------------

Best portability:
- Debian
- Ubuntu Server
- Rocky Linux
- AlmaLinux

Debian Host Setup
-----------------

Install virtualization packages:

sudo apt update

sudo apt install qemu-kvm \
libvirt-daemon-system \
virt-manager \
ovmf \
bridge-utils \
qemu-utils

Verify virtualization support:

egrep -c '(vmx|svm)' /proc/cpuinfo

Optional:

kvm-ok

Networking Recommendation
-------------------------

Use Linux bridge networking instead of NAT.

Example bridge:
- br0

Advantages:
- Cleaner ESXi migration
- Better network portability
- Easier server-style deployment

qcow2 to VMDK Conversion
------------------------

Install conversion tools:

sudo apt install qemu-utils

Convert qcow2 to VMDK:

qemu-img convert -f qcow2 \
-O vmdk source.qcow2 target.vmdk

This creates:
- VMware-compatible VMDK disks

Recommended ESXi Import Workflow
--------------------------------

Preferred Method:
1. Convert qcow2 to VMDK
2. Upload VMDK to ESXi datastore
3. Create a new empty VM in ESXi
4. Match virtual hardware:
   - UEFI
   - q35
   - Similar CPU/RAM
5. Attach existing VMDK

Advantages:
- Simplest workflow
- Fewer compatibility problems
- Easier troubleshooting
- Avoids OVF metadata issues

------------------------------

Do not use:

- PCIe passthrough

- Hypervisor-specific hardware features

Reason:

- Complicates migration

-------------------------------

    ↓

Summary
-------

Best overall workflow:

1. Build VM on Debian using KVM/QEMU
2. Use conservative portable hardware settings
3. Store master disk as qcow2
4. Convert to VMDK using qemu-img
5. Import manually into ESXi
6. Use OVF/OVA only when appliance packaging is needed

This provides:
- Maximum portability
- Linux-native management
- ESXi compatibility
- Easier backups
- Lower operational complexity
- Long-term maintainability

SSH Access Lessons From Test 1 - 2026-05-14
-------------------------------------------
For future VM shuttle tests, SSH access is a required end-of-test validation, not an optional convenience. The VM must run openssh-server, listen on TCP/22, and accept login for the expected lab user before the test is considered functionally complete.

Required guest-side actions:

  sudo apt update
  sudo apt install -y openssh-server qemu-guest-agent open-vm-tools cloud-init
  sudo mkdir -p /run/sshd
  sudo install -d -m 0755 /etc/ssh/sshd_config.d
  printf 'PasswordAuthentication yes\nKbdInteractiveAuthentication yes\n' | sudo tee /etc/ssh/sshd_config.d/99-codex-password-auth.conf
  sudo sshd -t
  sudo systemctl enable --now ssh qemu-guest-agent
  sudo systemctl enable --now open-vm-tools || true

Required validation before shutdown/export:

  virsh domifaddr VM_NAME --source agent
  # Ignore loopback addresses such as 127.0.0.1. Record the LAN address, for example 192.168.86.x.
  ping -c 3 VM_LAN_IP
  nc -vz -w 2 VM_LAN_IP 22
  ssh greg@VM_LAN_IP 'hostname; ip -br addr; id; systemctl is-active ssh qemu-guest-agent open-vm-tools'

If SSH fails after an offline edit, check for malformed sshd drop-ins. A literal string like `yes\nKbdInteractiveAuthentication` in one line is invalid. The file must contain two real lines.

## Precheck Update - 2026-05-15

The SSH-access workflow requires host-side validation tools in addition to the KVM/libvirt build tools. On kg55, the precheck found `sshpass` and `nmap` missing; they were installed and should be considered required for future VM shuttle runs.

Install/check prerequisites:

```bash
sudo apt update
sudo apt install -y sshpass nmap
command -v sshpass nmap virsh virt-install qemu-img virt-customize cloud-localds nc ip bridge brctl sha256sum curl git
```

For a clean second test, use a new VM name such as `vm-shuttle-test02` if `vm-shuttle-test01` is still defined or `/srv/vms/vm-shuttle-test01` still exists. Do not assume an old VMDK contains post-export SSH repairs; regenerate the VMDK from the corrected VM disk after shutdown when those fixes matter.

## Test 2 Cloud-Init Correction - 2026-05-15

Do not include `cloud-init status --wait` inside cloud-init `runcmd`; it waits on the same final stage that is executing it. For VM shuttle tests, run `cloud-init status --long` or `cloud-init status --wait` only from an external SSH validation command after the guest is reachable.

## Root Password For Lab `su -` - 2026-05-15

If a test VM must allow `su -`, cloud-init must set the root password explicitly. `su -` uses the root password, not the greg user password. For this lab workflow, set root to the same temporary lab password as greg in cloud-init `chpasswd`, then validate over SSH before export. Do not use this shared-password pattern for production.
